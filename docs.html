RecruiterOS documentation

RecruiterOS is the operating system for modern recruiting — a campaign-centric command center where every revenue activity lives in one place. This page documents the whole application: the screens that make up the UI, how the system is architected, and everything you need to run it locally or ship it to production.

Overview

The product is built around a single idea: the campaign is the atomic unit of work. Inside every campaign, five pillars run in sequence — Signals catch the trigger, Targets & Enrich build the list, Outreach sends the multi-channel sequence, Conversations handles replies, and Reporting measures what worked. Two revenue engines — Recruiting and Business Development — run on one shared infrastructure.

The codebase has three runnable layers, each independent:

LayerWhat it isRuns on
PortalThe static site — every .html page plus assets/. The full UI. No build step.Any static host / dev server
Backendintegration/ — a Next.js app that owns auth, data, webhooks, rate limits, and provider adapters.Node — localhost:3000
EdgeThe Chrome extension/ + bridge/ that perform LinkedIn actions safely from the user's own session.Browser + Node relay :8787
Demo vs. real. Most of the portal runs on realistic mock data so the experience can be explored with zero setup. The Outreach Studio ships a real engine (persistence, scheduler, safety limits, simulation clock), and the integration/ backend is the production path that wires in live providers.

Quick start

The fastest way to see the product. No dependencies required.

# Option A — just open the landing page
start index.html

# Option B — serve the folder with the zero-dependency dev server
node server.cjs
# → http://localhost:5173/  (clean URLs: /command, /alfred, /docs …)

# Option C — Windows one-liner that launches the server and opens the Studio
.\START-STUDIO.ps1

Then explore the app in this order:

  1. Open Command Center → Signals → Build campaign from signal.
  2. Prospects → run a search → Enrich cells, or add an ＋ AI column.
  3. Select rows → Add to sequence → review the AI-drafted Outreach.
  4. Open Outreach Studio → build a sequence → Enroll leads → use the sim clock (+1 day, +1 week) and watch connections accept, messages send, and replies land in the Inbox.
  5. Response → reply to a conversation and watch the AI classify intent.

Inside Outreach Studio & Campaign Studio

These two are the heart of execution. The Outreach Studio (alfred.html) is the most complete surface in the app:

  • Sequence — build a multi-step drip (connect → wait → message → email…) across LinkedIn / Email / X. Edit any step with a live, personalized preview.
  • Leads — add or import prospects, select, and Enroll them into a sequence.
  • Sim clock — +1 day / +1 week / +30 days buttons advance time so you can watch the campaign drip forward without waiting.
  • Inbox — replies land here and auto-pause that prospect's sequence.
  • Analytics — live funnel, accept/reply rates, activity by channel.
  • Settings — per-account daily caps, warm-up ramp, working hours, weekend pause, random delays, blacklist.
The core safety rule: a connection-request step gates its follow-ups — messages only fire after the invite is accepted. This is enforced in the engine and unit-tested. Default limits are conservative: ~20 connects/day, 30→50 messages, <100 views, a pending-invite cap, and a 14-day warm-up ramp from 30%→100%.

Architecture

The three layers communicate over a small set of seams designed to be swappable:

┌─────────────────────────────────────────────────────────────┐
│  PORTAL  (static *.html + assets/)                          │
│  the entire UI · runs in any browser · no build step        │
└───────────────┬─────────────────────────────────────────────┘
                │  fetch() →  config.js points at the backend
                ▼
┌─────────────────────────────────────────────────────────────┐
│  BACKEND  (integration/ — Next.js)   localhost:3000         │
│  auth · campaigns · prospects · response inbox · signals    │
│  billing · provider adapters (Unipile / Telnyx / simulated) │
└───────┬───────────────────────────────────────┬─────────────┘
        │ cloud provider (Unipile)               │ internal provider
        │ — fully server-side                    ▼
        │                              ┌────────────────────────┐
        │                              │  BRIDGE  (:8787)        │
        │                              │  per-account queues     │
        │                              └──────────┬─────────────┘
        │                                  poll / report
        ▼                                         ▼
   external APIs                       ┌────────────────────────┐
   (LinkedIn, email, SMS)              │  EXTENSION (Chrome MV3) │
                                       │  acts in the user's     │
                                       │  own LinkedIn session   │
                                       └────────────────────────┘

Outreach can run in one of three modes, selected per-account by RECRUITEROS_OUTREACH_PROVIDER: simulated (default — zero account risk), unipile (cloud, fully server-side), or internal (the bridge + browser extension). The same campaign code drives all three — only the adapter changes. See INTEGRATION.md for the full model.

The Alfred engine

The outreach automation engine lives in assets/js/alfred/. It is the same code in the browser (window.Alfred) and under Node (require), which is what lets it lift to a backend later without a rewrite. 

assets/js/alfred/
├─ alfred-core.js   UMD engine — data model, store, limits, channel
│                   adapters, personalization, scheduler, analytics
├─ alfred-ui.js     browser controller for the Studio
├─ backend.js       backend integration hooks
├─ studio-bridge.js postMessage bridge to the extension
└─ alfred.test.cjs  Node test harness — 30 assertions

Storage, channel adapters, and the clock are pluggable seams. Today it runs on localStorage + a simulated adapter; swap in a DB and real LinkedIn/Email/X adapters server-side and the engine code doesn't change.

Run the engine tests (Node is the only requirement, and only for tests):

node assets/js/alfred/alfred.test.cjs   # → 30 passed, 0 failed

Running the static site

The portal needs nothing but a browser. Three ways to serve it:

# zero-dependency dev server (recommended — gives you clean URLs)
node server.cjs                 # → http://localhost:5173
PORT=8080 node server.cjs       # override the port

# or just open the file
start index.html

# or any static server you like
python -m http.server 8000

server.cjs serves the current folder and resolves extension-less clean URLs (/alfred → alfred.html), so links match production.

Running the backend (integration/)

The Next.js app in integration/ is the production server. Its build step syncs the root *.html pages and assets/ into integration/public/ (via sync-public.cjs), so a single Next.js process serves both the UI and the API. 

cd integration
npm install
npm run build      # syncs *.html + assets into public/
npm run start      # → http://localhost:3000

node smoke.cjs     # end-to-end smoke test
Data persistence. Out of the box the backend uses in-memory reference repositories under integration/lib/db/. Swap these for a real database to persist across restarts. With no provider keys set, outreach and SMS default to the simulated provider.

Bridge & browser extension

The internal outreach provider runs LinkedIn actions from the user's own browser session — account-native and safe. Two parts:

  • Bridge (bridge/outreach-bridge.cjs) — a small relay on port 8787. The backend enqueues actions (/connect, /message, /inmail, /voice, …); the extension polls (/agent/poll) and reports results (/agent/report); accept/reply events are forwarded to the backend webhook (/agent/event). Queues are in-memory by design — swap for Redis in production.
  • Extension (extension/) — a Chrome MV3 extension. A background service worker holds the queue, limiter, and scheduler; a content script scrapes profiles and executes actions. SAFE_MODE is on by default (simulated). extension/config.js is the seam to edit: backend URL, daily caps, pacing, working hours.
cd bridge
cp .env.example .env          # set OUTREACH_TOKEN, AGENT_TOKEN, BACKEND_WEBHOOK_*
node outreach-bridge.cjs      # → http://localhost:8787  (GET /health)
node bridge.test.cjs          # end-to-end self-test

Load the extension via chrome://extensions → Load unpacked → select extension/. See bridge/README.md and extension/README.md for the full model.

Deployment

Two supported paths:

Static-only (the portal)

Drop the folder on any static host — Vercel (vercel or drag-and-drop), Netlify (app.netlify.com/drop), or GitHub Pages. Point your domain at it. Good for the marketing site and the demo experience.

Full app (portal + backend) on Hetzner

The production target is https://recruitersos.co on a Hetzner box, with Caddy terminating HTTPS in front of the Dockerized Next.js app. The repo includes Dockerfile, docker-compose.yml, Caddyfile, deploy.sh, and an auto-deploy watcher (auto-deploy.sh / install-auto-deploy.sh) that pulls and redeploys on push.

Follow the guides in order: HETZNER-SETUP.md → GO-LIVE-WALKTHROUGH.md (DNS + Caddy) → DEPLOY.md.

API

The backend exposes a REST API under /api; the full contract is in integration/openapi.yaml and browsable in the Developer Console. Representative routes:

GET/api/accountsThe team's LinkedIn accounts

GET/api/campaignsList / create campaigns
GET/api/response/listUnified inbox of classified replies
POST/api/linkedin/webhookInbound events (accepts, replies)
POST/api/sms/sendSend an SMS
POST/api/linkedin/cronCadence engine — server-side rate-limited execution
